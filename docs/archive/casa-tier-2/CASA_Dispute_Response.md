## Summary

Following the initial DAST scan, we have remediated all findings that were within our application's control. The following findings are being disputed because they relate to **infrastructure managed by our hosting provider (Vercel)** or **third-party SDKs (Clerk authentication)** and cannot be resolved through application code changes.

For each disputed finding, we provide the CWE ID, the evidence URL from the report, our classification, and a detailed technical explanation.

## Finding #4 – Proxy Disclosure

| Field | Value |
|-------|-------|
| **Severity** | Low |
| **CWE ID** | 204 |
| **Classification** | Infrastructure – Not Actionable |
| **Evidence URL** | `https://fine-grain-access-control-git-9654ff-kenyesh-gmailcoms-projects.vercel.app` |

### Response

The application is deployed on **Vercel's managed serverless platform**. The TRACE and OPTIONS method handling, as well as server identity headers, are controlled by Vercel's edge network infrastructure and cannot be modified by the application developer. This is a known, documented characteristic of all applications hosted on Vercel's platform.

**Mitigations applied:** We have suppressed the `X-Powered-By: Next.js` response header (which was within our application's control) by setting `poweredByHeader: false` in our Next.js configuration. The remaining "Vercel" identification in TRACE/OPTIONS responses is exclusively managed at the infrastructure level by Vercel and is not configurable by application owners.

## Finding #5 – Sub Resource Integrity Attribute Missing

| Field | Value |
|-------|-------|
| **Severity** | Low |
| **CWE ID** | 345 |
| **Classification** | Third-Party SDK – Not Actionable |
| **Evidence URLs** | `/dashboard` – `<script src="https://pumped-quetzal-63.clerk.accounts.dev/npm/@clerk/clerk-js@6/dist/clerk.browser.js" ...>` |
| | `/dashboard` – `<link rel="preload" href="https://pumped-quetzal-63.clerk.accounts.dev/npm/@clerk/ui@1/dist/ui.browser.js" ...>` |

### Response

The flagged `<script>` and `<link>` elements are **dynamically injected at runtime by our authentication provider, Clerk** (https://clerk.com). These tags are generated by the Clerk SDK (`@clerk/nextjs` npm package) via the `<ClerkProvider>` React component and are **not authored by our application code**.

Adding SRI (Subresource Integrity) hashes to these elements is not possible for the following reasons:

1. **Dynamic versioning:** The script URLs contain semver ranges (`@clerk/clerk-js@6`) that resolve to the latest minor/patch release on each page load. The content hash changes with every Clerk SDK update, making static SRI hashes immediately invalid.
2. **Runtime injection:** The `<script>` and `<link>` tags are injected by the Clerk SDK's initialization logic, not by our HTML templates. We do not have the ability to add `integrity` attributes to tags we do not render.
3. **Industry-standard pattern:** This CDN-hosted, auto-updating SDK pattern is standard practice for managed authentication providers (comparable to Auth0's Lock widget, Firebase Auth, and Okta's Sign-In Widget). None of these providers support SRI due to the same versioning constraints.

**Mitigations in place:** The Clerk SDK is loaded exclusively from Clerk's own managed CDN domain (`*.clerk.accounts.dev`), which is protected by TLS and Clerk's own security infrastructure. Our application enforces HTTPS-only communication.

## Finding #9 – Storable but Non-Cacheable Content

| Field | Value |
|-------|-------|
| **Severity** | Info |
| **CWE ID** | 524 |
| **Classification** | Correct Security Behavior – No Action Required |

### Response

The `no-store` Cache-Control directive observed on our HTML pages (`/`, `/dashboard`, `/setup`, `/robots.txt`, `/sitemap.xml`) is **intentional and represents the correct security posture**. These pages contain dynamic, user-specific content (authentication state, personalized dashboard data) and should not be cached by intermediate proxies or CDNs.

The scanner's observation that content is "storable but non-cacheable" is confirming that our caching policy is working as designed — responses are not being cached, which prevents sensitive user data from leaking through shared caches.

Static assets (`_next/static/chunks/*.js`, `*.css`) correctly use `max-age=31536000` for performance as they are content-hashed and contain no user-specific data.

## Finding #10 – Retrieved from Cache

| Field | Value |
|-------|-------|
| **Severity** | Info |
| **CWE ID** | 525 |
| **Classification** | Infrastructure CDN Behavior – Benign |

### Response

The `Age: 0` header observed in responses is a standard HTTP/1.1 header emitted by **Vercel's global CDN edge network**. An `Age: 0` value indicates the content was freshly fetched from the origin server (not served from a stale cache), which is the expected behavior for dynamic pages using `no-store` cache policies.

The scanner correctly notes that "the content was retrieved from a shared cache" — this refers to Vercel's edge infrastructure, which acts as a reverse proxy for all hosted applications. The `Age: 0` value confirms no stale data is being served. This is a benign infrastructure characteristic and does not represent a security risk.

For our public static files (`/skills/*.md`), we have explicitly set `Cache-Control: public, max-age=0, must-revalidate` to ensure these files are always revalidated, further confirming deliberate cache management.

### Response

The scanner tested our application with various fuzzed User-Agent strings (mobile browsers, search engine crawlers, etc.) and found **no differences in response behavior**.
