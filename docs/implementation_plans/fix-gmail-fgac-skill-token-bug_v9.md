# Root URL vs API Endpoint: Empirical Findings & Consistency Plan

## Summary

I ran actual tests against a local HTTP capture server and our proxy. Here are the **empirical facts**, not assumptions.

## Test Results

### Node.js (`googleapis` v171.4.0) — Local HTTP Capture Server

| Option | Sent to local server? | URL path received |
|--------|----------------------|-------------------|
| `rootUrl: 'http://localhost:9876/'` | ✅ YES | `/gmail/v1/users/me/labels` |
| `rootUrl: 'http://localhost:9876/api/proxy/'` | ✅ YES | `/gmail/v1/users/me/labels` (**path stripped!**) |
| `apiEndpoint: 'http://localhost:9876/'` | ❌ NO — went to googleapis.com | N/A |
| `apiEndpoint: 'http://localhost:9876/api/proxy/'` | ❌ NO — went to googleapis.com | N/A |
| `api_endpoint: 'http://localhost:9876/'` | ❌ NO — went to googleapis.com | N/A |

> [!CAUTION]
> **`apiEndpoint` and `api_endpoint` do NOT exist in the Node.js `googleapis` npm package.** They are silently ignored. Only `rootUrl` works, but it strips the path — the sdk always sends to `<origin>/gmail/v1/...` regardless of any path in the rootUrl.

### Why rootUrl strips the path (from source code)

`googleapis-common/build/src/apirequest.js` line 129-131:
```javascript
const originalUrl = new URL(options.url);                          // e.g. http://localhost/api/proxy/gmail/v1/users/me/labels
const path = originalUrl.href.substr(originalUrl.origin.length);   // = /api/proxy/gmail/v1/users/me/labels
options.url = new URL(path, parameters.context._options.rootUrl).href; // new URL('/api/proxy/...', 'http://localhost/api/proxy/')
```
`new URL` with an absolute path (starting with `/`) **replaces** the base's path, so: `new URL('/api/proxy/gmail/v1/...', 'http://localhost/api/proxy/')` = `http://localhost/api/proxy/gmail/v1/...` — wait, this SHOULD work. But the actual concat in `gmail/v1.js` uses `rootUrl + '/gmail/v1/...'` which, after the regex cleanup, produces a URL with origin `http://localhost:9876`. The `apirequest.js` rewrite then extracts `/gmail/v1/...` and does `new URL('/gmail/v1/...', rootUrl)` which resolves to just `http://localhost:9876/gmail/v1/...` — **the path in rootUrl IS dropped because `/gmail/v1/...` is an absolute path.**

### Python (`google-api-python-client`) — Source Code Analysis

From `discovery.py` lines 561, 575-576:
```python
base = urllib.parse.urljoin(service["rootUrl"], service["servicePath"])
# base = "https://gmail.googleapis.com/gmail/v1/"

if client_options.api_endpoint:
    base = client_options.api_endpoint  # REPLACES the entire base
```

Method URLs are constructed as: `base + method_path` (e.g., `base + users/me/labels`)

## The Correct Configuration Per Language

Based on the empirical evidence, here's what each SDK needs:

| Language | Parameter | Value | Final URL |
|----------|----------|-------|-----------|
| **Python** | `client_options={"api_endpoint": ...}` | `https://fgac.ai/api/proxy/gmail/v1` | `https://fgac.ai/api/proxy/gmail/v1/users/me/labels` ✅ |
| **Node.js** | `rootUrl: ...` | `https://fgac.ai/api/proxy/` ← (path gets stripped) | `https://fgac.ai/gmail/v1/users/me/labels` (needs subdomain routing) |
| **cURL** | Header + URL | `https://fgac.ai/api/proxy/gmail/v1/users/me/labels` | Direct ✅ |

> [!IMPORTANT]
> The Node.js SDK **always strips the rootUrl path**. There is no way to make it send to `/api/proxy/gmail/v1/...` via `rootUrl`. The only options are:
> 1. **Subdomain routing**: `rootUrl: 'https://gmail.fgac.ai/'` → middleware rewrites from `/gmail/v1/...` to `/api/proxy/gmail/v1/...`
> 2. **Local dev middleware rewrite**: Match `/gmail/v1/*` and rewrite to `/api/proxy/gmail/v1/*` (current approach)
